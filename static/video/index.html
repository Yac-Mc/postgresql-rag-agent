Demo técnica — postgresql-rag-agent

Recorrido en video del proyecto: arquitectura, flujo del grafo y decisiones técnicas clave.

Paseo técnico por el código

Estructura de carpetas relevante:

src/agent/
  graph.py          — orquestación del flujo con LangGraph
  nodes/            — nodos del grafo (seguridad, RAG, generación SQL, validación, ejecución, respuesta)
  api.py            — expone el agente como API REST (POST /chat), auth por API Key
  state.py          — esquema de estado compartido entre nodos

openspec/
  changes/          — propuestas, specs, diseño y tareas de cada cambio (SDD)
  specs/            — specs maestras vigentes

static/
  index.html        — slides del proyecto
  video/index.html  — esta página

El grafo de LangGraph (src/agent/graph.py) orquesta el flujo como un pipeline de nodos independientes:

analizar_seguridad → buscar_contexto_rag → generar_sql → validar_sql → ejecutar_consulta → generar_respuesta

Cada nodo es una función testeable por separado, no un único prompt monolítico.

Seguridad en 2 capas antes de ejecutar cualquier SQL:

1. Filtro de keywords bloqueadas (DROP, DELETE, TRUNCATE, ...)
2. Análisis semántico con Gemini (¿la intención de la pregunta es legítima?)
3. validate_sql: validación estructural del SQL generado

Ningún LLM ejecuta SQL directo contra la base — siempre hay una capa determinística entre la generación y la ejecución.

Manejo honesto de errores: el agente distingue un rechazo de seguridad (400) de un error transitorio del LLM como rate limit o timeout (503), para que el usuario final entienda si el sistema dijo "no" o simplemente falló temporalmente.
